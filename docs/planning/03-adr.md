## ADR-001: 실행 방식 - 글로벌 데몬

### 상태
**결정됨** (2026-04-16)

### 배경
Claude Code 실행 과정을 모니터링하는 spyglass의 인스턴스 관리 방식 결정 필요

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 프로젝트별 인스턴스** | 각 프로젝트마다 독립적인 spyglass 서버 | 완전한 격리, 프로젝트별 설정 가능 | 여러 프로젝트 동시 실행 시 포트 충돌, 리소스 중복 |
| **B. 글로벌 데몬** | 시스템에 한 번 설치, 모든 프로젝트 공유 | 데이터 통합, 리소스 효율성 | 별도 프로세스 관리 필요 |
| **C. 온디맨드** | 필요할 때만 실행 | 가장 가벼움 | 세션 간 데이터 유실 |

### 결정
**B. 글로벌 데몬** 채택

### 이유
1. **데이터 통합**: 모든 프로젝트의 세션을 한 곳에서 조회 가능
2. **리소스 효율**: 여러 인스턴스 중복 실행 방지
3. **프로젝스 관리**: `spyglass start` / `spyglass stop` 명령어로 간단히 관리

### 대안 채택 시 영향
- 프로젝트별 인스턴스 선택 시: 포트 관리 복잡성, 데이터 분산
- 온디맨드 선택 시: 세션 기록 누락, 장기적 분석 불가

## ADR-002: 데이터 저장소 - SQLite (WAL 모드)

### 상태
**결정됨** (2026-04-16)

### 배경
토큰 사용량 데이터를 효율적으로 저장하고 조회할 저장소 선택 필요

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. JSONL 파일** | Append-only 텍스트 파일 | 단순함, human-readable | 쿼리 성능 낮음, 동시성 문제 |
| **B. SQLite** | 경량 임베디드 DB | SQL 쿼리, 단일 파일, WAL 모드 지원 | 복잡한 쿼리 시 성능 제한 |
| **C. PostgreSQL/MySQL** | 외부 DB 서버 | 고성능, 동시성 | 별도 서버 설치 필요, 오버엔지니어링 |

### 결정
**B. SQLite (WAL 모드)** 채택

### 이유
1. **zero-config**: 별도 서버 설치 불필요
2. **SQL 쿼리**: 집계, 필터링, 정렬 용이
3. **WAL 모드**: 읽기/쓰기 동시성 확보
4. **ccflare 호환**: 기존 패턴 재사용 가능

### 설정
```sql
PRAGMA journal_mode = WAL;
PRAGMA busy_timeout = 5000;
PRAGMA synchronous = NORMAL;
```

## ADR-003: UI 프레임워크 - Ink (React for Terminal)

### 상태
**결정됨** (2026-04-16)

### 배경
터미널 기반 실시간 대시보드 구현을 위한 UI 프레임워크 선택

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. blessed-contrib** | Node.js 터미널 위젯 라이브러리 | 차트/테이블 내장, 가벼움 | React 미지원, 레거시 API |
| **B. Ink** | React for Terminal | React 개발자 친화, 컴포넌트 재사용 | 번들 크기 증가 |
| **C. stdout 갱신** | console.log 덮어쓰기 | 가장 가벼움 | 복잡한 UI 구현 어려움 |

### 결정
**B. Ink** 채택

### 이유
1. **ccflare 호환**: 동일한 스택으로 코드 재사용
2. **React 패턴**: 컴포넌트 기반 개발, 상태 관리 용이
3. **생태계**: useInput, useStdout 등 풍부한 훅 지원
4. **유지보수**: React 개발자가 쉽게 기여 가능

### 대안 검토
- blessed-contrib: 차트 기능은 유용하나 React 미지원으로 패스
- stdout 갱신: MVP 이후 고려 (가장 가벼운 폴리필)

## ADR-004: 데이터 수집 방식 - 훅 기반 (프록시 없음)

### 상태
**결정됨** (2026-04-16)

### 배경
Claude Code의 실행 데이터를 수집하는 방식 결정

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 프록시 서버** | ANTHROPIC_BASE_URL 설정으로 중간 가로채기 | 모든 데이터 자동 수집 | 설정 필요, 인증서 문제, 지연 발생 |
| **B. 훅 기반** | Claude Code 훅에서 명시적 전송 | 설정 불필요, 가벼움 | 데이터 누락 가능성 (훅 미등록 시) |

### 결정
**B. 훅 기반** 채택

### 이유
1. **zero-config**: 사용자가 ANTHROPIC_BASE_URL 설정할 필요 없음
2. **가벼움**: 프록시 오버헤드 없음 (<10ms)
3. **명시적**: 훅 등록을 통해 opt-in 방식
4. **ccflare 차별화**: 프록시 방식과 다른 접근

### 구현
```json
// settings.json
{
  "hooks": {
    "UserPromptSubmit": [{
      "hooks": [{
        "type": "command",
        "command": "bash spyglass-collect.sh prompt",
        "async": true,
        "timeout": 1
      }]
    }]
  }
}
```

## ADR-005: 알림 임계값 - 고정값 (10K 토큰)

### 상태
**결정됨** (2026-04-16)

### 배경
토큰 누수 감지를 위한 알림 기준 설정

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 고정값** | 단일 요청 10K 토큰 초과 시 알림 | 단순, 예측 가능, 구현 용이 | 사용자별 패턴 차이 무시 |
| **B. 동적 임계값** | 사용자 과거 패턴 기반 | 개인화된 알림, 오탐 감소 | 구현 복잡, 학습 기간 필요 |

### 결정
**A. 고정값 (10K 토큰)** 채택 (MVP)

### 이유
1. **MVP 적합**: 가장 빠르게 구현 가능
2. **직관적**: "한 요청에 10K는 많다"는 공감대
3. **Phase 2**: 사용자 설정 가능하도록 확장 예정

### 향후 개선
```typescript
// Phase 2에서 고려
const threshold = userSettings.threshold ?? 10000;
// 또는 동적: avgLast7Days * 2.0
```

## ADR-006: 개발 언어/런타임 - Bun + TypeScript

### 상태
**결정됨** (2026-04-16)

### 배경
spyglass 개발을 위한 언어와 런타임 선택

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. Node.js + TypeScript** | 표준 JavaScript 런타임 | 생태계 풍부, 안정성 | 느린 시작 시간, 번거로운 설정 |
| **B. Bun + TypeScript** | 최신 JavaScript 런타임 | TypeScript 네이티브, 빠른 실행 | 상대적으로 신생, 일부 호환성 이슈 |
| **C. Go** | 컴파일 언어 | 빠른 실행, 단일 바이너리 | 학습 곡선, ccflare와 다른 스택 |
| **D. Rust** | 시스템 프로그래밍 | 최고 성능, 안전성 | 느린 개발 속도, 복잡성 |

### 결정
**B. Bun + TypeScript** 채택

### 이유
1. **ccflare 호환**: 동일한 스택으로 코드/패턴 재사용
2. **개발 속도**: TypeScript 네이티브 지원, ts-node 불필요
3. **번들링**: 내장 번들러로 배포 용이
4. **성능**: Node.js 대비 빠른 시작 시간

### 검증
```bash
bun --version  # 1.2.8+
bun init       # TypeScript 프로젝트 초기화
bun run dev    # 개발 서버
```
