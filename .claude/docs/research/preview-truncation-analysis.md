# Preview 메시지 Truncation 문제 분석 및 해결 방안

## 문제 개요

### 사용자 보고 증상
Claude Spyglass 웹 대시보드에서 브라우저에 표시되는 preview message가 중간에 잘리는 문제:

```
오후 07:38 · 방금    prompt
◉
user    —    검색 뱃치 고도화가 필요합니다. 최근 요청 로그 페이지 처럼 각 클릭시 검색 키워드 뱃지 색상을
일부로 구분…    —    —    —    0ms    900722d2-6ec…
```

메시지가 "일부로 구분…"에서 끊겨 있으며, 사용자는 이것이 버그로 인식.

### 원인 분석

**현재 저장 구조:**
- `preview` 컬럼: 서버에서 100자로 잘라서 저장 (`collect.ts` line 370)
- `payload` 컬럼: 원본 JSON 그대로 저장 (truncate 없음)

**저장 시점의 동작:**
```typescript
// packages/server/src/collect.ts (line 363-376)
function extractPreview(payload: CollectPayload): string | null {
  if (payload.request_type !== 'prompt') return null;
  if (payload.payload) {
    try {
      const raw = JSON.parse(payload.payload) as Record<string, unknown>;
      const text = typeof raw.prompt === 'string' ? raw.prompt : null;
      if (text) return text.slice(0, 100);  // ← 저장 시점에 100자로 잘림
    } catch { /* ignore */ }
  }
  return null;
}
```

## 1. 프론트엔드 개발자 분석

### 현재 방식의 문제점

| 문제 | 설명 |
|------|------|
| **영구적 데이터 손실** | 100자 이후 내용은 DB에 저장되지 않아 복구 불가 |
| **이중 truncating** | 서버(100자) → 프론트(60자)로 중복 잘림 |
| **검색 불가** | 100자 이후 내용은 프론트 검색에서 제외 |
| **유연성 부족** | 페이지별 다른 길이 제한 적용 불가 (이미 잘려있음) |
| **데이터 불일치** | `preview` 컬럼과 `payload`의 prompt 필드가 다를 수 있음 |

**토큰 비용 산정과의 관계:**
- 토큰 비용 계산은 `preview` 컬럼을 사용하지 않음
- `tokens_input`, `tokens_output` 등 별도 필드 사용 → preview 길이와 무관

### 프론트엔드 처리 방식의 장단점

**장점:**
- **데이터 보존**: 원문 그대로 DB 저장 → 향후 활용 가능
- **유연성**: 페이지별 다른 길이 제한 적용 가능 (60자, 80자, 전체 등)
- **일관성**: `payload`에서 직접 추출 → 데이터 소스 단일화
- **검색 가능**: 원문 기반 클라이언트 사이드 검색 가능

**단점/고려사항:**
- **네트워크 오버헤드**: 긴 prompt가 많으면 응답 크기 증가 (gzip으로 완화)
- **메모리 사용**: `_promptCache`가 이미 전체 텍스트 캐싱 중
- **구현 변경 필요**: `extractPromptText()`, `contextPreview()` 수정

### 권장 방식

| 단계 | 항목 | 권장 내용 |
|------|------|-----------|
| **DB 저장** | preview 컬럼 | 원문 그대로 저장 (또는 1000~2000자 상한선만) |
| **프론트 표시** | truncating 시점 | 렌더링 시점에 페이지별로 동적 적용 |
| **페이지별 길이** | 요청 로그 | 60자 (현재 유지) |
| | 플랫 뷰 | 60자 (현재 유지) |
| | 턴 뷰 | 80~100자 (확대 권장) |
| | 확장 박스 | 전체 텍스트 |

**구현 우선순위:**
1. **Phase 1 (즉시)**: `renderers.js` 수정 - `extractPromptText()`가 `payload`에서 직접 추출하도록 변경
2. **Phase 2 (선택)**: 서버 수정 - `extractPreview()` 상한선 확대 (100 → 1000자)
3. **Phase 3 (선택)**: 페이지별 길이 적용 - 턴 뷰 등에 다른 `maxLen` 전달

## 2. UX/디자이너 분석

### 현재 UX 문제

#### 인지적 불일치 (Cognitive Dissonance)
- 사용자는 "왜 메시지가 100자에서 끊기는가?"라는 의문을 반복적으로 경험
- Claude가 실제로 처리한 메시지와 화면에 표시된 메시지 간 괴리 발생
- 토큰 비용 산정의 불투명성에 대한 불신

#### 맥락 상실 (Context Loss)
- 중요한 정보가 100자 경계에서 잘리는 경우가 빈번
- 예시:
  - "사용자가 요청한 기능을 구현하되, 보안상..." → 보안 제약 조건 누락
  - "이 버그를 수정해주세요. 문제는..." → 실제 문제 설명 누락

#### 신뢰도 저하
- 데이터의 불완전성이 시스템 전체에 대한 불신으로 확산

### 대안 방식 비교

| 방식 | 사용자 제어 | 유연성 | 시스템 상태 가시성 | 오류 예방 |
|------|-------------|--------|-------------------|-----------|
| **DB 원문 + 프론트 제한** | 높음 | 높음 | 높음 | 높음 |
| **현재 방식 유지** | 없음 | 없음 | 낮음 | 낮음 |

### 페이지별 최적 길이 제안

| 페이지 | 목적 | 권장 길이 | 표시 방식 |
|--------|------|-----------|-----------|
| **최근 요청 로그** | 빠른 스캐닝, 패턴 인식 | 80-120자 | 한 줄 또는 두 줄 + "더 보기" |
| **턴 뷰** | 대화 흐름 파악, 맥락 이해 | 200-300자 또는 3-4줄 | expand-box로 전체 토글 |
| **플랫 뷰** | 테이블 형태 전체 요약 | 60-80자 | 툴팁 또는 사이드 패널로 전문 |
| **상세 뷰** | 완전한 정보 확인 | 무제한 | 원문 그대로 표시 |

### UI 개선 제안

#### "더 보기/접기" 인터랙션
```
[미리보기 텍스트는 여기까지...]  [더 보기 ▼]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[확장 시 전체 내용이 여기에 표시됩니다]
[접기 ▲]
```

#### 점진적 공개 (Progressive Disclosure)
1. **초기 상태**: 2줄만 표시 (약 100-150자)
2. **"더 보기" 클릭**: 나머지 내용 슬라이드 다운
3. **상태 유지**: 세션 동안 확장 상태 기억

#### 스마트 트렁케이션
- 단어 경계에서 자르기 (word-break)
- 문장 경계 우선 (마침표, 물음표 기준)
- 마지막 완전한 문장까지만 표시

## 최종 권장사항

### 정책명: "원본 보존 + 상황별 표시"

#### 데이터 계층
- **DB**: 원문 그대로 저장 (무제한 또는 2000자 상한선)
- **API**: 원문 반환
- **Frontend**: 표시 로직에서 페이지별 제한

#### 표시 규칙 매트릭스
