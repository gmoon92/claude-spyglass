## ADR-001: 스킬 범위 — 에이전트 지침 vs 실행 도구

### 상태
**결정됨** (2026-04-19)

### 배경
데이터 분석 스킬의 목적을 어떻게 설정할지 결정해야 합니다. 스킬이 단순히 Claude에게 컨텍스트를 제공하는 지침 문서인지, 아니면 실제로 쿼리를 실행하거나 DB를 직접 조작하는 도구로 동작해야 하는지가 핵심입니다.

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 지침 문서형** | SKILL.md에 스키마·패턴·워크플로우 기술. Claude가 이를 바탕으로 코드 작성 | 즉시 구현 가능, 범용적 | 실제 쿼리 실행 불가 |
| **B. MCP 도구형** | SQLite MCP 서버 연동, 실제 쿼리 실행 가능 | 데이터 직접 확인 | MCP 서버 구성 필요, 인증/권한 복잡 |
| **C. 혼합형** | 지침 문서 + Bash 도구로 `sqlite3` CLI 실행 | 분석 시 실제 데이터 확인 | 로컬 DB 경로 의존, 환경별 차이 |

### 결정
**옵션 A (지침 문서형)**

스킬은 Claude에게 프로젝트 데이터 구조에 대한 완전한 컨텍스트를 제공하는 에이전트 지침 문서로 설계한다. 실제 쿼리 실행은 Claude가 코드를 작성하고 사용자가 실행하는 방식을 유지한다.

### 이유

1. **아키텍트 관점**: 스킬의 역할은 "알고 있어야 할 것"을 Claude에게 제공하는 것. MCP 연동은 별도 feature(data-analyst-mcp)로 분리해야 관심사 분리 원칙에 맞음.
2. **데이터 엔지니어 관점**: 현재 DB 경로가 `~/.spyglass/spyglass.db`이지만 `SPYGLASS_DB_PATH` 환경변수로 변경 가능. 경로를 하드코딩한 MCP 연동은 취약함.
3. **DX 엔지니어 관점**: 지침 문서형은 즉시 효과를 발휘. 스킬 등록만으로 이후 모든 데이터 관련 대화에서 패턴 오류가 줄어듦.

## ADR-002: 문서 구조 — 단일 SKILL.md vs 참조 문서 분리

### 상태
**결정됨** (2026-04-19)

### 배경
스킬 문서의 정보량이 많습니다. 테이블 스키마, 마이그레이션 이력, 집계 쿼리 예시, 데이터 흐름, 작업 워크플로우를 하나의 파일에 담을지, 핵심과 레퍼런스를 분리할지 결정해야 합니다.

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 단일 SKILL.md** | 모든 내용을 하나의 파일에 | 파일 하나만 관리 | 너무 길어져 핵심이 묻힘 |
| **B. SKILL.md + references/schema.md** | 핵심 워크플로우는 SKILL.md, DDL·쿼리는 별도 파일 | 역할 분리, 각 파일이 적절한 길이 | 파일 수 증가 |
| **C. SKILL.md + 다수 references** | 토픽별 파일 분리 (schema.md, queries.md, migration.md 등) | 세부 탐색 용이 | 관리 부담 증가, 분산된 정보 |

### 결정
**옵션 B (SKILL.md + references/schema.md)**

- `SKILL.md`: 트리거 조건, 데이터 흐름, 테이블 요약, 마이그레이션 패턴 개요, 작업 워크플로우 (에이전트가 매 대화에서 참조할 핵심 내용)
- `references/schema.md`: 전체 DDL, PRAGMA 설정, 마이그레이션 코드 패턴, 집계 쿼리 예시 (필요 시 참조할 상세 레퍼런스)

### 이유

1. **DX 엔지니어 관점**: `ui-designer` 스킬도 동일한 패턴(`SKILL.md` + `references/design-system.md`) 사용. 프로젝트 내 일관성 유지.
2. **아키텍트 관점**: 워크플로우와 레퍼런스는 수정 빈도가 다름. 스키마가 변경되면 `schema.md`만 업데이트. 워크플로우가 변경되면 `SKILL.md`만 업데이트.
3. **데이터 엔지니어 관점**: `schema.md`에 실제 DDL을 두면 신규 마이그레이션 추가 시 참조 문서이자 검증 체크리스트 역할을 동시에 수행.

## ADR-003: 스킬 트리거 설계 — 키워드 기반 vs 의도 기반

### 상태
**결정됨** (2026-04-19)

### 배경
Claude Code 스킬은 `description` 필드의 트리거 조건에 따라 자동 선택됩니다. 트리거를 너무 넓게 잡으면 관련 없는 요청에서 불필요하게 스킬이 로드되고, 너무 좁게 잡으면 실제 필요한 상황에서 스킬이 누락됩니다.

### 고려한 옵션

| 옵션 | 트리거 방식 | 예시 |
|------|------------|------|
| **A. 광범위 키워드** | "데이터", "DB", "SQL" 등 일반 용어 | 과잉 트리거 가능 |
| **B. 작업 유형 명시** | "테이블 추가", "컬럼 추가", "마이그레이션", "쿼리 최적화" | 의도가 명확한 요청에서 정확히 매칭 |
| **C. 도메인 + 작업 조합** | 스토리지 도메인 + 작업 유형 조합 | 가장 정밀하나 트리거 문구가 길어짐 |

### 결정
**옵션 B (작업 유형 명시)**

트리거 키워드: "데이터 분석", "테이블 추가", "컬럼 추가", "쿼리 최적화", "스키마 변경", "마이그레이션 추가", "통계 개선", "데이터 흐름 분석"

### 이유

1. **DX 엔지니어 관점**: 다른 스킬들도 동일한 패턴 사용 (예: `ui-designer`는 "디자인 개선", "UI 바꿔줘"). 일관된 트리거 설계.
2. **아키텍트 관점**: 작업 유형 기반 트리거는 사용자의 의도를 직접 반영하므로 오탐이 적음.
3. **데이터 엔지니어 관점**: "데이터" 단독은 너무 광범위. 구체적 작업 키워드가 명확한 시그널.

## ADR-004: 마이그레이션 가이드 위치 — SKILL.md 내부 vs references 분리

### 상태
**결정됨** (2026-04-19)

### 배경
신규 마이그레이션 추가 방법은 SKILL.md에 포함해야 할지, `references/migration.md`로 분리해야 할지 결정이 필요합니다.

### 결정
**SKILL.md 내부에 패턴 요약 포함 + references/schema.md에 코드 예시 포함**

마이그레이션은 데이터 분석 에이전트가 가장 자주 다루는 작업 중 하나이므로, 변경 순서와 패턴은 SKILL.md 본문에 직접 기술한다. 전체 코드 예시는 `references/schema.md`에 포함한다.

### 이유

1. 마이그레이션 패턴은 새 컬럼/테이블 추가 워크플로우의 핵심. 별도 파일로 분리하면 에이전트가 매번 파일을 열어야 함.
2. `schema.md`에 코드 패턴을 포함하면 DDL과 마이그레이션 코드를 함께 볼 수 있어 맥락이 명확함.
3. ADR-003에서 결정한 B+C 구조(`SKILL.md` + `references/schema.md`)와 일관됨.

## 전문가 이견 및 해소

### 이견 1: 실제 쿼리 실행 필요성

**데이터 엔지니어**: 분석 스킬이 실제 DB를 조회할 수 없으면 반쪽짜리.  
**아키텍트**: Bash 도구로 `sqlite3` CLI를 사용하면 충분한 경우가 많음. 복잡한 MCP 연동은 오버엔지니어링.  
**해소**: 현재는 지침 문서형으로 시작. 실사용 후 MCP 연동 필요성이 확인되면 `data-analyst-mcp` 별도 feature로 분리.

### 이견 2: 스킬 vs CLAUDE.md 직접 기술

**DX 엔지니어**: CLAUDE.md에 데이터 관련 지침을 직접 써도 되지 않나?  
**아키텍트**: CLAUDE.md는 전역 지침용. 도메인별 상세 컨텍스트를 CLAUDE.md에 모으면 파일이 비대해지고 모든 대화에서 불필요하게 로드됨.  
**해소**: 스킬 방식 유지. 데이터 관련 요청에서만 선택적으로 로드.

### 이견 3: 스키마 문서 자동 동기화

**데이터 엔지니어**: `schema.ts`가 변경될 때 `references/schema.md`를 자동으로 업데이트하는 훅이 필요하다.  
**아키텍트**: 자동화는 좋지만 현재 팀 규모와 복잡도를 고려하면 과잉. 마이그레이션 추가 시 수동으로 schema.md도 업데이트하는 것이 현실적.  
**해소**: 수동 업데이트 정책 유지. `data-analyst` 스킬의 검증 체크리스트에 "schema.md 업데이트 여부 확인" 항목 포함.
