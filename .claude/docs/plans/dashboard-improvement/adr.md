## ADR-001: source 컬럼 소급 기본값 — NULL vs 문자열 상수

### 상태
**결정됨** (2026-04-18)

### 배경

T-01에서 `requests.source` 컬럼을 추가할 때, 기존 레코드(마이그레이션 전 수집된 데이터)에 어떤 값을 부여할지 결정해야 한다. 원래 계획은 `DEFAULT 'claude-code-hook'`이었으나 세 전문가 모두 이에 의문을 제기했다.

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A | `DEFAULT 'claude-code-hook'` | 쿼리 시 null 처리 불필요 | 실제 unknown 레코드와 실제 hook 레코드를 구분 불가, 통계 오염 |
| B | `DEFAULT NULL` + UI에서 `null → '알 수 없음'` | 의미론적으로 정확, 데이터 무결성 보장 | UI에서 null 방어 코드 필요 |

### 결정

`ALTER TABLE requests ADD COLUMN source TEXT` — 기본값 없이 추가하여 기존 레코드는 NULL, 신규 레코드는 훅이 전송한 값을 저장한다.

### 이유

1. (백엔드/아키텍트 관점) 마이그레이션 전 데이터가 실제로 `claude-code-hook`에서 수집됐다는 보장이 없으므로, 가짜 출처 값을 부여하면 소스별 통계가 오염된다.
2. (프론트엔드 관점) UI에서 `source ?? '알 수 없음'`으로 표시하는 비용이 통계 정확성 손실보다 훨씬 작다.
3. NULL이 "알 수 없음"을 표현하는 가장 표준적인 DB 패턴이다.

### 대안 채택 시 영향

- 옵션 A 선택 시: 소급 레코드가 `claude-code-hook` 출처로 보이므로 향후 소스 필터링/집계 결과가 왜곡된다.

## ADR-002: colspan 관리 전략 — 상수화 vs 하드코딩 유지

### 상태
**결정됨** (2026-04-18)

### 배경

항목 A(버그)에서 이미 확인된 것처럼, `togglePromptExpand()`에 `colspan="5"`가 하드코딩되어 플랫 뷰(4컬럼)에서 overflow가 발생했다. T-05/T-06에서 컬럼이 각각 4→6, 5→7로 늘어나면 동일 버그가 재발할 위험이 있다.

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A | 상수 선언 (`const FLAT_VIEW_COLS = 6`) | 컬럼 수 변경 시 한 곳만 수정 | 단일 HTML 파일에 상수 선언 영역 추가 필요 |
| B | 하드코딩 수정만 | 빠름 | 다음 컬럼 추가 시 동일 버그 재발 |

### 결정

각 뷰마다 컬럼 수 상수를 JS 변수로 선언하고, `colspan` 속성값과 모든 expand/empty 행에서 해당 상수를 참조한다.

```js
const FLAT_VIEW_COLS = 6;      // 시각 | 타입 | 툴 | IN | OUT | 응답시간
const RECENT_REQ_COLS = 7;     // 시각 | 타입 | 툴 | IN | OUT | 응답시간 | 세션 ID
```

### 이유

1. (세 전문가 공통 의견) A 항목 버그가 이미 발생한 패턴이므로 단순 수정은 기술 부채를 반복한다.
2. 단일 HTML 파일 구조에서도 상수화는 10줄 이내의 변경으로 구현 가능하다.
3. 향후 컬럼 추가 시 누락 오류를 원천 차단한다.

### 대안 채택 시 영향

- 옵션 B 선택 시: 이번 수정은 동작하지만 다음 컬럼 추가 시 동일 버그가 재발한다.

## ADR-003: duration_ms 포맷팅 — 공통 유틸 함수 vs 인라인 분기

### 상태
**결정됨** (2026-04-18)

### 배경

T-04(요약 카드), T-05(플랫 뷰), T-06(최근 요청), T-07(턴 뷰)에서 동일한 포맷 규칙(`< 1000ms → {N}ms`, `≥ 1000ms → {N.N}s`, `0 → —`)이 반복 적용된다. 인라인으로 각 위치에 구현하면 규칙이 4곳에 분산된다.

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A | `formatDuration(ms)` 유틸 함수 선언 | 단일 진실 원천, 규칙 변경 시 한 곳만 수정 | 함수 위치 합의 필요 |
| B | 각 렌더링 함수 내 인라인 처리 | 즉시 구현 가능 | 4곳에 중복 로직 분산 |

### 결정

`formatDuration(ms)` 함수를 `index.html` JS 유틸 섹션에 단독 선언하고, T-04~T-07 구현 시 이 함수를 공통으로 참조한다.

```js
function formatDuration(ms) {
  if (!ms || ms <= 0) return '—';
  if (ms < 1000) return `${ms}ms`;
  return `${(ms / 1000).toFixed(1)}s`;
}
```

### 이유

1. (세 전문가 공통 의견) 동일 포맷 로직이 4개 렌더링 함수에 분산되면 미래 수정 비용이 선형으로 증가한다.
2. T-04가 가장 먼저 구현되므로, T-04에서 유틸 함수를 선언하고 이후 태스크에서 참조하는 순서가 자연스럽다.

## ADR-004: promptPreview 중복 구현 통합

### 상태
**결정됨** (2026-04-18)

### 배경

항목 B/C에서 확인된 것처럼:
- `promptPreview()` (725행): `prompt` 타입만, 60자 제한
- 턴 뷰 inline (962행): 모든 타입, 80자 제한

두 구현이 독립적으로 존재하여 동작이 다르다.

### 고려한 옵션

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A | `promptPreview(r, maxLen=60)` 시그니처 변경 후 단일화 | 중복 제거, maxLen 파라미터로 유연성 확보 | 턴 뷰 코드 리팩터링 필요 |
| B | 두 구현 유지, 각각 수정 | 변경 범위 최소 | 드리프트 지속 |

### 결정

`promptPreview(r, maxLen = 60)` 시그니처로 변경하고, `prompt` 타입 전용 가드를 제거하여 모든 타입의 payload를 미리보기 가능하게 확장한다. 턴 뷰 inline 코드는 이 함수 호출로 교체한다.

### 이유

1. (프론트엔드/백엔드 관점) 중복 구현을 유지하면 세 번째, 네 번째 구현이 생기는 드리프트가 반복된다.
2. `maxLen` 파라미터로 호출처마다 길이를 조정할 수 있어 기존 80자 요구사항도 만족한다.
3. B 항목(타입 가드 제거)와 C 항목(중복 제거)를 동시에 해결한다.

### 대안 채택 시 영향

- 옵션 B 선택 시: T-05 수정 이후에도 턴 뷰에 세 번째 미리보기 구현이 잔존한다.

## ADR-005: H 항목(avg_tokens 미렌더링) 처리 방침

### 상태
**결정됨** (2026-04-18)

### 배경

`renderTools()`에서 API가 반환하는 `avg_tokens`를 화면에 표시하지 않는 버그(H 항목)가 발견됐으나, 원래 계획의 T-01~T-08에 대응 태스크가 없었다. 세 전문가 모두 이 누락을 지적했다.

### 결정

이번 개선 범위에 **포함**한다. T-09로 추가하여 `renderTools()` 함수에서 `avg_tokens`를 표시하도록 수정한다. 변경 범위가 작아(단일 함수 내 렌더링 추가) 다른 태스크와 충돌 없이 독립적으로 진행 가능하다.

### 이유

1. API가 이미 `avg_tokens`를 반환하므로 백엔드 변경 없이 프론트엔드만 수정하면 된다.
2. 전문가 세 명이 모두 누락으로 지적한 항목을 "다음 단계"로 미루는 것은 데이터 일관성 목표에 반한다.

## ADR-006: avgDurationMs 집계 범위

### 상태
**결정됨** (2026-04-18)

### 배경

백엔드 엔지니어가 `AVG(duration_ms)` 쿼리에 시간 범위 조건이 없어, 장기 운영 시 초기 느린 응답이 평균을 왜곡할 수 있다고 지적했다.

### 결정

초기 구현은 **전체 기간 평균**으로 구현한다. 단, API 파라미터(`?window=Nd`)를 위한 인터페이스를 열어두어 향후 기간 선택 기능 추가 시 API 변경 없이 대응 가능하도록 설계한다.

### 이유

1. YAGNI 원칙에 따라 현재 요구사항(전체 평균 표시)에 집중한다.
2. 파라미터 인터페이스를 열어두는 것은 코드 추가 없이 API 명세 수준의 합의로 충분하다.
3. 운영 데이터가 충분히 쌓인 후 기간 필터 UI를 추가하는 것이 현실적인 순서다.
