# 원자성 태스크 분해 가이드

## 핵심 원칙

**하나의 태스크 = 하나의 커밋**이며, 모든 커밋은 다음 조건을 만족해야 합니다:

1. **독립 빌드 가능**: 이 커밋만 체크아웃해도 빌드/실행 가능
2. **테스트 통과**: 기존 테스트가 깨지지 않음
3. **git bisect 지원**: 버그 발생 커밋을 `git bisect`로 정확히 찾을 수 있음
4. **단일 관심사**: 하나의 논리적 변경만 포함

## tasks.md 형식

```markdown
# {Feature} 작업 목록

> 기반 문서: plan.md, adr.md  
> 작성일: YYYY-MM-DD  
> 총 태스크: {N}개

---

## 태스크 분해 패턴

### 계층별 분해 (권장)

의존성이 낮은 계층부터 위로 쌓아 올립니다:

```
1. 데이터 모델/스키마 정의
2. 저장소 계층 (DB, 파일)
3. 비즈니스 로직/서비스 계층
4. API/인터페이스 계층
5. UI/클라이언트 계층
6. 테스트 (각 계층 완료 직후)
7. 통합 테스트
8. 문서화
```

### 테스트 태스크 배치 규칙

- 단위 테스트: 해당 구현 태스크 **바로 다음**에 배치
- 통합 테스트: 관련 모듈 구현 완료 **후** 별도 태스크로
- 테스트 없는 구현 태스크는 **피합니다**

```
T-01: UserRepository 구현 (feat)
T-02: UserRepository 단위 테스트 (test)  ← 바로 뒤에 배치
T-03: AuthService 구현 (feat)
T-04: AuthService 단위 테스트 (test)
T-05: Auth 통합 테스트 (test)            ← 관련 모듈 완료 후
```

### 너무 큰 태스크 분해

태스크가 4시간을 넘거나 파일 수정이 5개 이상이면 분해합니다:

```
나쁜 예:
T-01: 전체 사용자 인증 시스템 구현 (16h)

좋은 예:
T-01: User 테이블 스키마 및 모델 정의 (1h)
T-02: UserRepository CRUD 구현 (2h)
T-03: 비밀번호 해시 유틸리티 구현 (1h)
T-04: AuthService 로그인/로그아웃 구현 (2h)
T-05: JWT 토큰 발급/검증 구현 (2h)
T-06: Auth 미들웨어 구현 (1h)
T-07: 인증 API 엔드포인트 구현 (2h)
T-08: 인증 E2E 테스트 (2h)
```

### 너무 작은 태스크 통합

개별 상수나 타입 정의처럼 1-2줄 변경은 묶습니다:

```
나쁜 예:
T-01: ERROR_CODE 상수 추가
T-02: ErrorType enum 추가
T-03: ApiError 타입 추가

좋은 예:
T-01: 에러 처리 타입 정의 (ErrorCode, ErrorType, ApiError)
```

## 커밋 타입 가이드

| 타입 | 사용 시점 |
|------|----------|
| `feat` | 새 기능 구현 |
| `test` | 테스트 추가/수정 |
| `fix` | 버그 수정 |
| `refactor` | 기능 변경 없는 코드 개선 |
| `docs` | 문서 추가/수정 |
| `chore` | 빌드, 설정, 의존성 변경 |
| `perf` | 성능 개선 |

## 고위험 태스크 표시

DB 마이그레이션, 인증 변경, 외부 API 연동 등 롤백이 복잡한 태스크에는
`⚠️ 고위험` 태그를 붙이고 롤백 절차를 상세히 기술합니다:

```markdown
## T-12: DB 스키마 마이그레이션 ⚠️ 고위험

### 롤백 방법
```bash
# 마이그레이션 롤백
bun run db:migrate:rollback

# 또는 직접 되돌리기
sqlite3 ~/.spyglass/db.sqlite "ALTER TABLE ..."
```

**주의**: 데이터 손실 가능성 있음. 프로덕션 적용 전 백업 필수.
```
