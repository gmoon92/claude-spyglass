# cc:refactor — Claude Code 문서 리팩터링

LLM(모델)이 지시를 오해 없이 정확히 실행할 수 있도록 Claude Code 메타 문서의 모호성을 제거하고 개선합니다.

---

## 실행 순서

> 아래 6단계를 반드시 이 순서대로 실행합니다. 각 단계는 이전 단계의 결과를 사용합니다.

### Step 1. 인수 타입 판별 및 대상 파일 읽기

인수 타입을 아래 순서로 판별한 후 처리합니다.

| 타입           | 판별 기준                           | 처리 방법                                                                 |
|--------------|---------------------------------|-----------------------------------------------------------------------|
| **파일 경로**    | `.md`로 끝나거나 경로 구분자(`/`)를 포함한 패턴 | `Read` 도구로 직접 읽음                                                      |
| **디렉토리 경로**  | 파일이 아닌 디렉토리를 가리키는 경로            | `Glob`으로 `${CLAUDE_PROJECT_DIR}/<디렉토리>/**/*.md` 목록 확인 후 각 파일을 순서대로 처리 |
| **프롬프트 텍스트** | 위 두 가지에 해당하지 않는 자연어             | 인수 텍스트에서 스킬명·파일명 키워드를 식별하여 `.claude/skills/<이름>/SKILL.md`, `CLAUDE.md`, `.claude/rules/` 등에서 대상 파일을 추론한 뒤 읽음 |

**공통 규칙**:

- 상대 경로는 `${CLAUDE_PROJECT_DIR}/<경로>`로 해석합니다.
- 여러 파일인 경우(디렉토리 또는 복수 파일): **파일별로 Step 2~6 전체를 순차 실행**합니다.
- **deep 모드 결정**: Step 6은 항상 실행합니다. 대상 파일 읽기 전에 `AskUserQuestion` 도구로 아래와 같이 질문합니다:

  ```
  question: "리팩터링 완료 후 품질 평가 방식을 선택해주세요."
  header: "평가 방식"
  options:
    - label: "빠른 검토"
      description: "수정된 파일만 빠르게 검토합니다."
    - label: "상세 비교 평가"
      description: "수정 전후 품질을 비교한 HTML 리포트를 생성합니다. (시간이 더 소요됩니다)"
  ```

    - "빠른 검토" 선택 → **빠른 검토 모드**로 진행
    - "상세 비교 평가" 선택 → **상세 비교 평가 모드**로 진행

### Step 2. 8가지 기준으로 문제 탐지 (모두 적용)

아래 8가지 기준을 **빠짐없이** 적용하고, 발견된 문제를 행 번호와 함께 Step 4 보고서 작성용 내부 목록으로 유지합니다.

#### 기준 A: 모호한 지시 탐지

다음 패턴을 찾으세요. 패턴에 해당하면 내부 목록에 추가합니다.

| 패턴           | 모호한 예              | 수정 방향                             |
|--------------|--------------------|-----------------------------------|
| 빈도·정도 부사 불명확 | "필요시", "적절히", "가끔" | 트리거 조건을 조건문(`~인 경우`, `~일 때`)으로 명시 |
| 범위 미정 동사     | "확인하세요", "처리하세요"   | 무엇을, 어떤 기준으로 확인·처리할지 명시           |
| 조건 없는 조건문    | "경우에 따라 X를 하세요"    | 경우의 조건을 구체적으로 열거                  |
| 암묵적 전제       | "기존 설정을 유지하세요"     | "기존 설정"이 가리키는 파일·키 이름을 명시         |
| 결과 기준 없는 지시  | "잘 작동하도록 수정하세요"    | 성공 기준(테스트 통과, 오류 없음 등)을 명시        |

#### 기준 B: 프로세스/워크플로우 순서 충돌 탐지

**적용 조건**: 단계 번호(1., 2., Step N), "먼저/그 다음/마지막으로", 또는 순서 있는 목록이 있는 문서에만 적용합니다.

확인 항목:

1. **전방 참조**: 아직 생성·정의되지 않은 파일·변수·상태를 참조하는 단계
2. **역방향 의존**: 단계 N이 단계 M(M > N)의 완료를 전제로 함
3. **상충 지시**: 단계 X에서 "항상 A를 하라" + 단계 Y에서 "A를 하지 말라"
4. **누락 단계**: 단계 N이 상태 S를 전제하지만, 상태 S를 만드는 단계가 없음

#### 기준 C: LLM 관점 명확화

사람은 자명하지만 LLM이 오해할 수 있는 표현을 찾으세요:

| 문제 유형    | 원문 예           | 수정 방향                                         |
|----------|----------------|-----------------------------------------------|
| 대명사 모호성  | "그것을 저장하세요"    | 주어·목적어를 구체적인 식별자(변수명, 파일명)로 교체                |
| 범위 없는 지시 | "현재 파일을 수정하세요" | "사용자가 마지막으로 언급한 파일을 수정하세요"로 명시                |
| 중의적 동사   | "정리하세요"        | 구체적 동작으로 분해 (예: "중복 항목 제거 후 알파벳 순 정렬")        |
| 묵시적 컨텍스트 | "설정을 적용하세요"    | 설정 파일 경로(`${CLAUDE_PROJECT_DIR}/…`)와 키 이름을 명시 |
| 출력 위치 미정 | "결과를 출력합니다"    | 파일 저장인지 화면 출력인지 명시                            |

#### 기준 D: 파일 경로 표준화

- **변환 대상**: 절대 경로(`/Users/…`, `/home/…`), 프로젝트 루트 기준 상대 경로(`.claude/`, `docs/`)
- **변환 방법**: `${CLAUDE_PROJECT_DIR}/` 를 앞에 붙임
    - 예: `.claude/settings.json` → `${CLAUDE_PROJECT_DIR}/.claude/settings.json`
- **변환 제외**: `~` (홈 디렉토리) 경로, 코드 블록(```) 내부 경로

#### 기준 E: 참조 대상 존재 검증

문서에서 참조하는 스킬 이름, 에이전트 유형, 파일 경로가 실제로 존재하는지 확인합니다.

**탐지 대상과 검증 방법**

| 참조 유형   | 문서 내 패턴 예                                     | 검증 방법                                                                  |
|---------|-----------------------------------------------|------------------------------------------------------------------------|
| 스킬 이름   | `Skill("name")`, `스킬: name`, `name 스킬`        | `Glob` 으로 `${CLAUDE_PROJECT_DIR}/.claude/skills/<name>/SKILL.md` 존재 확인 |
| 에이전트 유형 | `subagent_type: "name"`, `name 에이전트`          | 아래 기본 제공 목록과 대조                                                        |
| 파일 경로   | `${CLAUDE_PROJECT_DIR}/…`, `.claude/…`, `~/…` | `Glob` 또는 `Read` 로 경로 존재 확인                                            |

**기본 제공 에이전트 유형** (목록에 있으면 검증 불필요):
`general-purpose`, `Explore`, `Plan`, `backend`, `pm`, `claude-code-guide`, `statusline-setup`

**존재하지 않을 때 처리 규칙**

참조 대상이 없으면 수정 초안에서 직접 수정하지 않고 Step 4 보고서의 "참조 대상 검증 결과" 섹션에 사용자 안내로 출력합니다.

| 참조 유형   | 사용자 안내 내용                                                                                    |
|---------|----------------------------------------------------------------------------------------------|
| 스킬 이름   | "`${CLAUDE_PROJECT_DIR}/.claude/skills/<name>/SKILL.md` 가 없습니다. 스킬 이름을 확인하거나 스킬을 먼저 생성하세요."  |
| 에이전트 유형 | "에이전트 유형 `<name>`이 유효하지 않습니다. 유효한 유형: `general-purpose`, `Explore`, `Plan`, `backend`, `pm`" |
| 파일 경로   | "`<path>`가 존재하지 않습니다. 경로를 수정하거나 파일을 먼저 생성하세요."                                               |

#### 기준 F: 파일명 규칙 준수 여부 검증

**적용 조건**: 분석 대상이 `.claude/` 디렉토리 내 파일이거나, 디렉토리 전체를 분석하는 경우에만 적용합니다.

이 스킬의 분석 대상(SKILL.md, CLAUDE.md, 훅, 프롬프트, rules, 서브에이전트) 파일명이 아래 규칙을 따르는지 확인합니다.

| 문서 유형     | 위치                                                 | 파일명 패턴                                       | 올바른 예                  | 잘못된 예                 |
|-----------|----------------------------------------------------|----------------------------------------------|------------------------|-----------------------|
| CLAUDE.md | 프로젝트 루트 또는 `.claude/`                              | `CLAUDE.md` (대문자 고정)                         | `CLAUDE.md`            | `claude.md`           |
| rules 파일  | `.claude/rules/`                                   | kebab-case                                   | `planning-workflow.md` | `planningWorkflow.md` |
| 훅 스크립트    | `.claude/hooks/`                                   | kebab-case                                   | `post-tool-use.sh`     | `postToolUse.sh`      |
| 스킬 정의     | `.claude/skills/<name>/`                           | 진입점 `SKILL.md` (대문자 고정), 스킬 디렉토리명 kebab-case | `commit/SKILL.md`      | `Commit/skill.md`     |
| 서브에이전트    | `.claude/agents/` 또는 `.claude/agents/<directory>/` | kebab-case                                   | `backend.md`           | `backendAgent.md`     |
| 프롬프트      | `.claude/docs/archive/<feature>/prompts/`          | kebab-case                                   | `analyze-task.md`      | `analyzeTask.md`      |

**처리 규칙**: 파일명 규칙 위반이 발견되면 수정 초안에서 직접 수정하지 않고, Step 4 보고서의 "파일명 규칙 위반" 섹션에 아래 형식으로 출력합니다. 파일명 변경은 파일시스템 작업이므로 사용자가 직접
처리해야 합니다.

```
| # | 현재 파일명 | 위반 규칙 | 권장 파일명 |
|---|-----------|---------|-----------|
```

#### 기준 G: 에이전트 구조 원칙 준수

**적용 조건**: `.claude/agents/` 하위 `.md` 파일에만 적용합니다.

에이전트는 스킬이 `Task()`로 위임하는 "단일 전문 역할 서비스"입니다. 아래 항목 누락은 역할 경계 불명확, 비용 최적화 불가, 자체 검증 불가로 이어집니다.

| 탐지 항목                                           | 누락 시 문제                         |
|-------------------------------------------------|---------------------------------|
| `model:` frontmatter                            | 모든 호출이 세션 모델로 실행 → 비용·품질 최적화 불가 |
| 역할 경계 명시 ("NOT responsible for" 또는 책임 외 영역)     | 역할 침범 가능                        |
| `Success_Criteria` — 측정 가능한 완료 기준 (수치·패턴 포함)    | 완료 판단 불가                        |
| `Final_Checklist` — 완료 전 자체 검증 항목               | 자체 검증 불가                        |
| `Failure_Modes_To_Avoid` 또는 금지 사항               | LLM 과잉 창의성 억제 불가                |
| READ-ONLY 에이전트(분석·검토 전용)인데 `disallowedTools` 없음 | 실수로 파일 수정 가능                    |

READ-ONLY 에이전트 판별: 이름에 `analyst`, `architect`, `critic`, `reviewer` 포함 또는 description에 "분석", "검토", "평가" 전용 명시.

**처리 규칙**: 발견 항목은 수정 초안에서 직접 내용을 추가하지 않습니다 — 내용은 해당 에이전트의 도메인 지식이 필요합니다. Step 4 보고서의 "에이전트·스킬 구조 준수 검증" 섹션에 누락 항목을 안내합니다.

---

#### 기준 H: 스킬 구조 원칙 준수

**적용 조건**: `.claude/skills/` 하위 `SKILL.md` 파일에만 적용합니다.

스킬은 사용자가 직접 호출하는 "워크플로우 전체 조율자"입니다. 아래 항목 누락은 자율성 불명확, 사용 경계 모호, 무한 루프 위험으로 이어집니다.

| 탐지 항목                                                                    | 누락 시 문제          |
|--------------------------------------------------------------------------|------------------|
| `level:` frontmatter                                                     | Claude 자율성 힌트 없음 |
| `Do_Not_Use_When` 또는 사용 제외 케이스·대안 스킬                                     | 사용 경계 모호         |
| 오케스트레이션 스킬(`Skill()`/`Agent()` 2개 이상 순차 호출)인데 `pipeline:` frontmatter 없음 | 파이프라인 의도 미선언     |
| 반복 루프에 최대 횟수 미명시                                                         | 무한 루프 위험         |
| `Skill()`/`Agent()` 위임에 failsafe(실패 처리) 없음                               | 위임 실패 시 스킬 전체 중단 |
| `Final Checklist` — 완료 전 자체 검증                                           | 자체 검증 불가         |

**처리 규칙**: 기준 G와 동일 — 수정 초안 직접 수정 금지, Step 4 보고서에 안내.

> 기준 G·H의 상세 탐지 기준·판별 예시·처리 규칙 예시:
> `${CLAUDE_PROJECT_DIR}/.claude/skills/cc-docs-refactor/references/criteria-structure.md`

### Step 3. 수정 초안 작성

- Step 2에서 기록한 **모든** 문제를 반영한 수정 초안을 작성합니다.
- **원문의 지시 항목은 빠짐없이 유지합니다.** 명확화 과정에서 지시를 제거하거나 축소하지 마세요.
- 문제가 없는 부분은 변경하지 않습니다(과잉 수정 금지).

**토큰 절감 원칙** (수정 초안 작성 시 적용):

- 같은 내용의 반복 설명은 제거합니다.
- 산문(prose)보다 표·목록 형식을 선호합니다.
- 예시는 핵심 1~2개로 제한합니다(망라식 나열 금지).

#### frontmatter description 동기화 검토

`description`은 Claude Code 모델이 스킬 호출 여부를 결정하는 **유일한 트리거 메커니즘**입니다. 본문 내용이 아무리 좋아도 description이 부실하면 스킬이 호출되지 않습니다.

본문 수정 초안 작성 후, 아래 4가지 중 하나라도 해당하면 `description` 수정 초안도 함께 작성합니다:

| 변경 유형        | 판단 기준                                                        |
|--------------|--------------------------------------------------------------|
| 적용 범위 변경     | 대상 문서 유형(SKILL.md, CLAUDE.md 등)이 추가·제거된 경우                  |
| 주요 동작 변경     | 스킬의 핵심 출력물이나 동작 방식이 달라진 경우                                  |
| 트리거 조건 변경    | "언제 이 스킬을 사용하는지"에 해당하는 키워드·상황이 달라진 경우                       |
| 트리거 키워드 누락   | 본문에 새 사용 사례·기능이 추가됐는데, description에 해당 키워드·문구가 없는 경우 |

`description` 수정 시 유지할 형식:

- **앞부분**: 스킬이 하는 일 (무엇을, 어떻게) — 구체적인 입출력·범위 포함
- **중간**: 사용자가 실제로 입력할 법한 트리거 문구를 동의어·변형 포함하여 열거 (최소 3가지)
- **끝부분**: `"…요청할 때 반드시 이 스킬을 사용하세요"` 형식의 적극적 호출 안내

undertrigger 방지를 위해 description은 "pushy"하게 작성합니다. 사용자가 스킬 이름을 직접 언급하지 않아도 의도가 맞으면 트리거되도록 다양한 문구를 포함하세요.

#### 수정 초안 임시 저장

수정 초안 작성 완료 후 `Bash` 도구로 아래 경로에 저장합니다:

```
${CLAUDE_TMP_DIR}/benchmark/<target>/<YYYYMMDD>/draft.md
```

`<target>` 값 결정 (Step 6-1에서도 동일하게 사용):

| 대상 유형         | target 값                                                  |
|---------------|-----------------------------------------------------------|
| `SKILL.md` 파일 | 스킬 디렉토리명 (`commit/SKILL.md` → `commit`)                   |
| 일반 `.md` 파일   | 파일명 확장자 제외 (`planning-workflow.md` → `planning-workflow`) |
| 디렉토리          | basename (`.claude/rules/` → `rules`)                     |

이 파일은 tmp 디렉토리에 저장되므로 git에 추적되지 않습니다.

### Step 5. 사용자 승인 후 저장

1. **사용자 승인을 기다립니다.** 승인 전에 파일을 수정하지 마세요.
2. 승인 후, 수정 전 원본을 스냅샷으로 백업합니다:
    - 스냅샷 경로: `${CLAUDE_PROJECT_DIR}/.claude/archive/<target>/benchmark/<YYYYMMDD>/snapshot.md`
    - `<YYYYMMDD>`는 Step 3에서 사용한 날짜와 동일한 값을 사용합니다.
    - `Bash` 도구로 `mkdir -p <날짜디렉토리경로> && cp <원본파일경로> <스냅샷경로>` 실행
3. `Edit` 도구로 Step 3 수정 초안을 적용합니다.
    - 부분 승인 시: 승인된 항목만 적용합니다.
    - 전체 재작성이 필요한 경우에만 `Write` 도구를 사용합니다.

### Step 6. skill-creator 품질 검토

항상 실행합니다. Step 1에서 결정된 deep 모드에 따라 skill-creator에 전달하는 컨텍스트가 달라집니다.

| 모드       | 조건            | skill-creator 역할                      |
|----------|---------------|---------------------------------------|
| 빠른 검토    | "빠른 검토" 선택    | 수정된 파일 품질 검토 — baseline 비교·eval 루프 없음 |
| 상세 비교 평가 | "상세 비교 평가" 선택 | 스냅샷 vs 수정본 eval 루프 실행, HTML 리포트 생성    |

#### 6-1. eval 파일 경로 결정 (상세 비교 평가 모드인 경우에만)

빠른 검토 모드이면 이 단계를 건너뜁니다.

eval 파일은 아래 경로에 날짜별로 생성합니다:

```
${CLAUDE_TMP_DIR}/benchmark/<target>/<YYYYMMDD>/evals.json
```

`<target>` 은 Step 3에서 결정한 값을 재사용합니다.

#### 6-2. 기존 파일 초기화 및 이전 날짜 정리 (상세 비교 평가 모드인 경우에만)

빠른 검토 모드이면 이 단계를 건너뜁니다.

```bash
TODAY=$(date +%Y%m%d)
EVAL_DIR="${CLAUDE_TMP_DIR}/benchmark/<target>/${TODAY}"
EVAL_FILE="${EVAL_DIR}/evals.json"

# 오늘 디렉토리 생성
mkdir -p "${EVAL_DIR}"

# 파일이 이미 존재하면 내용 초기화
if [ -f "${EVAL_FILE}" ]; then
  echo '{"skill_name":"cc:refactor","evals":[]}' > "${EVAL_FILE}"
fi

# 오늘 날짜 이전 tmp 날짜 디렉토리 제거 (tmp/<target>/ 하위에서만, archive는 건드리지 않음)
find "${CLAUDE_TMP_DIR}/benchmark/<target>" \
  -maxdepth 1 -type d -name '????????' \
  ! -name "${TODAY}" -exec rm -rf {} +
```

#### 6-3. skill-creator 호출

`<target>`, `<YYYYMMDD>`, `<스킬명>`(대상 SKILL.md의 frontmatter `name:` 필드값) 은 Step 3과 Step 5에서 결정된 실제 값으로 치환한 뒤, `Skill("document-skills:skill-creator")`를 호출합니다.

**빠른 검토 모드인 경우:**

> 기존 스킬의 수정 결과를 검토해주세요. baseline 비교 없이 수정된 파일의 품질만 평가합니다.
> - 스킬 경로: `<수정된 파일이 속한 스킬 디렉토리>`
> - 수정된 파일: `<수정된 파일 경로>`
> - 목표: 모호성 제거·명확성 개선 여부에 대한 피드백. eval 루프와 HTML 리포트는 생략합니다.

**상세 비교 평가 모드인 경우:**

> 기존 스킬 개선 작업입니다.
> - 스킬 경로: `<수정된 파일이 속한 스킬 디렉토리>`
> - 이전 버전 스냅샷(baseline): `${CLAUDE_PROJECT_DIR}/.claude/archive/<target>/benchmark/<YYYYMMDD>/snapshot.md`
> - eval 파일: `${CLAUDE_TMP_DIR}/benchmark/<target>/<YYYYMMDD>/evals.json`
> - workspace 경로: `${CLAUDE_TMP_DIR}/benchmark/<target>/<YYYYMMDD>/workspace/`
> - old vs new 비교 평가를 실행하고, 평가 결과를 `${CLAUDE_PROJECT_DIR}/.claude/archive/<target>/benchmark/<YYYYMMDD>/eval-viewer.html`에 HTML 파일로
    저장합니다.
> - 목표: 수정 전후 품질 및 토큰 비용 비교

**상세 비교 평가 모드 완료 후:**

1. `Read` 도구로 `${CLAUDE_PROJECT_DIR}/.claude/archive/<target>/benchmark/<YYYYMMDD>/eval-viewer.html`을 읽어 평가 결과(점수 변화·개선 항목·권고사항)를 요약하여
   사용자에게 출력합니다.
2. eval-viewer.html의 전체 경로를 사용자에게 제공합니다.
3. 요약 결과를 바탕으로 추가 수정이 필요한지 사용자에게 확인합니다.

**위임 실패 시**: `Skill("document-skills:skill-creator")`를 호출할 수 없는 경우, "skill-creator 검토를 건너뜁니다" 메시지를 출력하고 스킬을 완료 처리합니다. 검토 없이 중단하지 않습니다.

**여러 파일 처리 시**: 각 파일의 Step 2~5를 모두 완료한 후, Step 6을 파일별로 독립 실행합니다.
